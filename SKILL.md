# repo-release — 일관된 GitHub 출시 워크플로우

자잘한 작업을 끝내고 GitHub에 올릴 때마다 손으로 반복하던 절차 — 레포를 같은 방식으로 만들고,
설명을 적고, 버전을 태그하고, 릴리스 노트를 쓰는 — 를 한 흐름으로 묶는다. 핵심 목표는 **일관성**
(레포가 매번 같은 컨벤션으로 만들어진다)과 **안전**(비밀키·개인정보가 절대 공개로 새어나가지 않는다)이다.

## 0. 원칙 — 이 스킬이 지키는 것

- **푸시·커밋은 사용자가 이 스킬을 발동했을 때만.** 이 스킬 자체가 명시적 트리거다. 그래도 실제로
  원격에 올리기 직전(§6)에는 무엇이 올라가는지 요약해 보여주고 최종 확인을 받는다 — GitHub 푸시는
  되돌리기 어렵고 공개 레포는 한번 인덱싱되면 삭제해도 캐시에 남기 때문이다.
- **비밀정보 스캔은 건너뛸 수 없는 게이트다(§2).** 발견 시 즉시 멈춘다. "아마 괜찮을 것"은 없다.
- **사실 vs 추론 구분.** 버전 bump 제안, description 초안 등은 모두 "초안 → 사용자 승인" 구조다.
  Claude이 멋대로 확정하지 않는다.
- **산문은 한국어, 코드·명령·경로·태그·커밋 프리픽스는 영어.**

## 1. 사전 점검 — 시작하면 먼저 한다

1. **작업 디렉토리 확인.** 어느 폴더를 올리는지 확정한다(기본은 현재 작업 디렉토리). 이 폴더명이
   레포명의 기본값이 된다.
2. **`gh` 인증 확인.** `gh auth status` 를 실행한다. 실패하면 사용자에게 `gh auth login` 을
   안내하고 멈춘다 — 인증 없이는 레포 생성·릴리스가 불가능하다.
3. **git 상태 파악.** 폴더가 git repo인지(`git rev-parse --is-inside-work-tree`), remote가
   있는지(`git remote -v`) 확인한다. 이걸로 신규/업데이트를 가른다(§3).
4. **현재 브랜치 확인.** `git branch --show-current`. 이 워크스페이스는 멀티 세션이라 엉뚱한
   브랜치에서 커밋·푸시하는 사고를 막아야 한다. 의도와 다르면 사용자에게 확인한다.

### 컨벤션 (매번 동일하게 적용)

- **계정**: `xixwxx` (GitHub).
- **레포명 = 로컬 폴더명**, `snake_case`. `_vN` 같은 버전 접미사 금지 — 버전은 git tag로 관리한다.
- **도구**: `gh` CLI (`gh repo create`, `gh release create`) + `git`.
- **라이선스**: 공개 레포에는 항상 **MIT**, 저작권자는 GitHub 핸들 **`xixwxx`** (실명 비노출).

## 2. 보안 스캔 — 푸시·생성 전 반드시 통과해야 하는 게이트

비밀키(API key, token, private key)나 개인정보가 공개 레포에 올라가면 즉시 사고다. 그래서 어떤
경로(신규/업데이트)든 **여기를 먼저 통과**해야 한다.

번들된 스크립트를 실행한다:

```bash
bash "<이 스킬 경로>/scripts/preflight_scan.sh"
```

스크립트 동작:
- **`gitleaks` 가 설치돼 있으면 우선 사용** (working tree + **git history 전체** 스캔). 전용 도구라
  커버리지가 가장 넓다.
- 없으면 **내장 정규식 fallback** 으로 스캔: OpenAI/Anthropic/AWS/GitHub/Google/Slack 키 패턴,
  `-----BEGIN ... PRIVATE KEY-----`, 추적 중인 `.env`/`credentials`/`*.pem`/`id_rsa` 류 파일,
  그리고 **절대경로에 박힌 사용자명**(`/Users/<username>/...`) — 개인 환경 노출.
- 스캔 범위는 자동 분기: **신규 레포면 올라갈 워킹트리 전체**, **업데이트면 이번에 커밋·푸시될
  변경분** + 추적 중인 위험 파일. 둘 다 가능하면 git history도 본다.

**발견 시**: 푸시를 **즉시 중단**하고, 어떤 파일 몇 번째 줄에서 어떤 패턴이 걸렸는지 사용자에게
보고한 뒤 처리를 함께 정한다:
- 아직 커밋 안 됨 → `.gitignore` 에 추가하거나 파일을 옮긴다.
- 이미 추적 중(`git rm --cached <file>`) → 인덱스에서 빼고 `.gitignore` 추가.
- 이미 커밋·푸시됨 → history 정리(`git filter-repo` 등) **그리고 해당 키를 즉시 회전(rotate)**.
  공개된 키는 이미 유출된 것으로 간주한다.

스캔이 깨끗할 때만 다음 단계로 간다. `gitleaks` 가 없으면 fallback이 돌았음을 사용자에게 알리고,
중요한 공개라면 `brew install gitleaks` 후 재스캔을 권한다(정규식 fallback은 커버리지가 더 낮다).

> 참고: 이 워크스페이스엔 `.env` **읽기**를 막는 secret-guard hook이 이미 있다. 이 스캔은
> **푸시 시점** 게이트라 서로 보완된다. 스캔 중에도 `.env` 내용을 화면에 출력하지 않는다.

## 3. 신규 vs 업데이트 자동 감지

- **remote 없음 / git repo 아님** → **신규 레포 경로(§4)**.
- **remote 있음** → **업데이트 경로(§5)**.

애매하면(remote는 있는데 동기화 안 됨 등) 사용자에게 상황을 보고하고 어느 쪽인지 확인한다.

## 4. 신규 레포 경로 — 처음 GitHub에 올리기

### 4.1 레포 구성요소 준비

폴더를 보고 아래를 챙긴다. **이미 있는 파일은 건드리지 않는다** — 기존 파일은 존중하고, 보강이
필요해 보이면 "이런 내용 추가하면 어때?" 라고 **초안만 따로 제안**한다(덮어쓰기 금지).

- **README.md** (없으면 한국어 초안 생성): 제목, 한 줄 설명, (해당 시) 설치/사용법, 라이선스 한 줄.
- **.gitignore** (없으면): 언어/스택 자동 감지(node/python/etc)해서 생성. `.env`·credentials·빌드
  산출물이 포함되도록 보장한다.
- **LICENSE** (없으면): **MIT**, `Copyright (c) <올해> xixwxx`.
- **정적 사이트/게임이면**(루트에 `index.html` 등) GitHub **Pages 활성화**까지 하고 배포 URL을
  사용자에게 알려준다.

### 4.2 공개 범위 확인 (생성 직전, 매번)

가시성은 한번 잘못 정하면 위험하므로 **기본 private** 로 두되, 만들기 직전에 "이 레포 public으로
공개할까, private로 둘까?" 를 **반드시 확인**한다. public을 고른 경우에만 LICENSE(§4.1)와 Pages가
의미를 갖는다.

### 4.3 description 초안 → 승인

레포 description(GitHub 우측 상단 한 줄 소개)을 코드 보고 **한국어 초안**으로 작성해 보여주고,
사용자 승인·수정 후 확정한다.

### 4.4 생성·첫 커밋·첫 릴리스

```bash
git init                       # git repo가 아니면
git add -A
git commit -m "<Conventional Commits 형식, §7>"
gh repo create xixwxx/<폴더명> --<public|private> --source=. --remote=origin \
  --description "<승인된 description>" --push
```

첫 릴리스는 **`v1.0.0`** 으로 태그하고 GitHub Release를 만든다(§8). Pages 대상이면 활성화한다.

## 5. 업데이트 경로 — 기존 레포에 변경 올리기

### 5.1 변경 분석 → 버전 bump 제안

`git status` / `git diff` 로 변경을 파악하고, **변경 규모를 보고 semver bump를 제안**한다.
판단은 Claude이 하되 **확정은 사용자가** 한다:

- 호환을 깨는 변경(공개 API/동작 변경, 제거) → **major** (`v2.0.0`)
- 기능 추가(하위호환) → **minor** (`v1.1.0`)
- 버그 수정·문서·내부 정리 → **patch** (`v1.0.1`)

직전 버전은 기존 태그에서 자동 계산한다: `git tag --list 'v*' --sort=-v:refname | head -1`.
태그가 하나도 없으면 첫 릴리스로 보고 `v1.0.0` 부터 시작한다.

bump 근거를 한 줄로 설명하고("공개 함수 시그니처가 바뀌어 major 제안"), 사용자가 OK하면 진행한다.

### 5.2 커밋 → 푸시

변경을 Conventional Commits(§7)로 커밋한다. README 등 기존 파일은 §4.1 규칙대로(덮어쓰기 금지,
보강은 제안만) 다룬다. 그다음 §6 안전 게이트를 거쳐 푸시한다.

## 6. 안전 게이트 — 원격에 올리기 직전

푸시·생성 직전에 멈추고 다음을 요약해 보여준 뒤 **최종 확인**을 받는다:

- 대상 레포·브랜치, public/private
- 이번에 올라가는 변경 요약 (`git diff --stat` / 새 파일 목록)
- 매길 버전 태그
- §2 보안 스캔 결과(통과 여부)

사용자가 명시적으로 OK한 뒤에만 `git push` / `gh repo create --push` / `gh release create` 를 실행한다.

## 7. 커밋 메시지 컨벤션

- **Conventional Commits 프리픽스** + **한국어 본문/요약**. Co-Authored-By 트레일러는 **붙이지 않는다**
  (사용자 단독 히스토리 유지).
- 프리픽스: `feat:`(기능) `fix:`(버그) `docs:`(문서) `chore:`(설정/잡일) `refactor:` `test:` `style:`.

**예시 1** — Input: 다크모드 토글을 추가함 → Output: `feat: 다크모드 토글 추가`
**예시 2** — Input: 점수가 0으로 표시되던 버그 수정 → Output: `fix: 점수 0 표시 버그 수정`
**예시 3** — Input: README에 설치법 보강 → Output: `docs: README 설치법 추가`

## 8. 버전 태그 + GitHub Release

semver(`vX.Y.Z`)로 **annotated tag** 를 달고 GitHub Release를 만든다:

```bash
git tag -a v1.2.0 -m "<릴리스 제목>"
git push origin v1.2.0
gh release create v1.2.0 --title "<제목>" --notes "<릴리스 노트>"
```

**릴리스 노트**는 직전 태그 이후 변경을 모아 **한국어 초안**으로 작성하고(`git log <이전태그>..HEAD
--oneline` 참고), 사용자 승인 후 확정한다. 변경을 추가/수정/버그수정 등으로 묶어 읽기 좋게 정리한다.

## 9. 마무리

- 올린 뒤 레포 URL(과 Pages URL이 있으면 그것도)을 사용자에게 알려준다.
- 진단·테스트로 만든 임시 파일이 있으면 정리한다(커밋된 코드는 유지).
- 사실만 보고한다: 무엇이 올라갔고, 어떤 태그·릴리스가 생겼는지. 스캔을 건너뛰었거나 실패한 단계가
  있으면 숨기지 말고 그대로 말한다.

## 빠른 흐름 요약

```
사전 점검(gh auth·폴더·브랜치)
  → 보안 스캔(게이트, 실패 시 중단)
  → 신규?  → 구성요소 준비 → public/private 확인 → description 초안·승인 → 생성·첫 커밋 → v1.0.0 Release
  → 업데이트? → diff 분석 → bump 제안·승인 → 커밋
  → 안전 게이트(요약 + 최종 확인)
  → 푸시 → 태그 → Release(노트 초안·승인)
  → URL 안내 · 정리
```
